September

First Week

During the first week, I explored the codebase and familiarized myself with the project's existing implementations.

Second Week

During this week, I developed and integrated simple evaluator, Python and JavaScript evaluator from Chainforge into PromptStudio.

  • Modified Chainforge's jsonToYml function to handle new evaluator cases.
  • Python evaluator implementation
    • Implemented execute_python: runs an evaluation script via a subprocess with a JSON wrapper to collect results.
  • Simple evaluator implementation
    • Generated JavaScript code for the simple evaluator using Chainforge's createJSEvalCodeFor.
    • And then run the Simple evaluator by reusing the existing execute_javascript function.
  • Integrated the evaluators into the application (file worker.ts).

Third Week

This week I added Chainforge processors to PromptStudio and tested the application on a cluster.

  • Adapted Chainforge's jsonToYml to support new processor types.
  • Database adjustments for processors
    • Modified the SQL script: added an optional format column for split processors and an optional code column for processors containing code.
    • Created a dedicated enum for processor types instead of reusing evaluator types.
  • Implemented the Split processor
    • Developed execute_split for the split processor using Chainforge's splitText function.
  • Implemented the Python processor
    • Implemented the Python processor by reusing execute_python.
  • Wired the new processor implementations into worker.ts.
  • Testing the application on a cluster
    • Cluster tests (madagh server of DIRO): opened a tunnel (hole punch) to expose MySQL from my PC to madagh, then validated functionality. The application behaved as expected on the cluster.

Fourth Week

This week I fixed the bug of the input token calculation for an experiment input.

  • Input token calculation logic Fix
    • Context of the bug: Input token calculation always return 0
    • Causes
      • Our token calculator function have an if statement to check for final_dataset_id, if it's null we skip the calculation.
      • But new experiments always have their final_dataset_id = null by default at the handle_save_template() function.
      • Existing experiments have their result already processed so the calculator function just skip the calculation and returns 0.
    • Solution
      • At the final_dataset_id check, we just go ahead and calculate the total input token.

October

First Week

This week I worked on implementing the LLM Evaluator and Multi-Evaluator features.

  • LLM Evaluator Implementation
    • Extended the jsonToYml flow to support LLM evaluators by introducing a code block for evaluators of type llm. I initially integrated this type into the existing evaluator architecture (JavaScript, Python, Simple) by storing LLM configuration inside the code attribute of the evaluator object.
    • Implemented execution logic for the LLM evaluator in worker.ts. The workflow consists of extracting values from promptVarsDict and using them to replace the {response} placeholder in the prompt template. And then call the queryLLM function to get the evaluation result.
  • Multi-Evaluator Implementation
    • Extended the jsonToYml flow to support multi-evaluators by introducing a code block for evaluators of type multieval. My initial approach was to bypass the multieval wrapper and directly execute its child evaluators independently, rather than treating them as a grouped structure. In this approach, instead of a flow like prompt → multieval, the execution becomes prompt → [child evaluators].
    • Each child evaluator was identified using a naming convention in the format {multieval-name}-{childCounter}, where the counter starts at 0 and increments for each child.
    • Implemented execution logic for multi-evaluators in worker.ts. The system retrieves child evaluators by matching name prefixes and executes them sequentially.

After discussion with professor Syriani, it was determined that the above approaches for both features are inefficient and architecturally incorrect. And better solutions will be integrate in the following week.

Second Week

This week I implemented the modifications proposed by Professor Syriani regarding the LLM Evaluator and Multi-Evaluator architectures.

  • Refactoring of jsonToYml Pipeline
    • A significant refactoring was performed on the jsonToYml pipeline. Previously, both YAML node construction and JSON parsing logic were coupled within a single function. This was refactored by introducing a dedicated buildYmlNode function responsible for YAML node construction, file extraction, and tracking unused nodes. The resulting structure returns the necessary information to the main jsonToYml function.
    • This refactoring was introduced to support recursive processing required for Multi-Evaluator nodes.
  • Adjustments to LLM Evaluator
    • A dedicated Llm_evaluator table was introduced to store LLM-specific evaluator configurations. This separation ensures a clear distinction between LLM evaluators and other evaluator types. In this design, Multi-Evaluators are persisted in both the Evaluator and Multi_evaluator tables, while LLM evaluators are linked to the Llm_Param table, which stores the configuration of the underlying LLM grader.
    • New functions (handle_save_llm_evaluator and save_evaluator) were implemented to manage the persistence workflow for LLM evaluators.
    • In worker.ts, LLM evaluators are now resolved using get_llm_evaluator_by_id, which retrieves the full configuration required for execution via queryLLM. The logic for constructing promptVarsDict was also adjusted to align with the updated evaluator structure.
  • Adjustments to Multi-Evaluator
    • The Multi-Evaluator generation logic in jsonToYml was updated to use recursive calls to buildYmlNode, replacing previous conditional branching logic.
    • A multievalChildToNodeFormat adapter was introduced to normalize child evaluators, which may differ structurally from standalone evaluators of the same type.
    • A dedicated Multi_evaluator table was introduced to store Multi Evaluators. Similar to the LLM evaluator design, Multi Evaluators are stored in both the Evaluator and Multi_evaluator tables to support both generic and specialized representations.
    • A Multi_evaluator_mapping table was added to store relationships between multi-evaluators and their child evaluators using (multi_evaluator_id, child_evaluator_id) pairs.
    • The execution logic in worker.ts was updated to support recursive evaluation, removing earlier implementations based on explicit conditional branching.
  • Update to Configuration Saving Logic
    • The save_config function in configHandler.ts was updated to include explicit handling for both LLM and Multi-Evaluator types, delegating their persistence to handle_save_llm_evaluator and handle_save_multi_evaluator respectively.
    • The handle_save_evaluator function was modified to return the generated node_id, enabling its reuse when constructing Multi-Evaluator mappings in the Multi_evaluator_mapping table.
