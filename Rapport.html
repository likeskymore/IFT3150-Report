September

First Week

During the first week, I explored the codebase and familiarized myself with the project's existing implementations.

Second Week

During this week, I developed and integrated simple evaluator, Python and JavaScript evaluator from Chainforge into PromptStudio.

  • Modified Chainforge's jsonToYml function to handle new evaluator cases.
  • Python evaluator implementation
    • Implemented execute_python: runs an evaluation script via a subprocess with a JSON wrapper to collect results.
  • JavaScript evaluator implementation
    • Generated JavaScript code for the simple evaluator using Chainforge's createJSEvalCodeFor.
    • Implemented the JavaScript evaluator by reusing the existing execute_javascript function.
  • Integrated the evaluators into the application (file worker.ts).

Third Week

This week I added Chainforge processors to PromptStudio and tested the application on a cluster.

  • Adapted Chainforge's jsonToYml to support new processor types.
  • Database adjustments for processors
    • Modified the SQL script: added an optional format column for split processors and an optional code column for processors containing code.
    • Created a dedicated enum for processor types instead of reusing evaluator types.
  • Implemented the Split processor
    • Developed execute_split for the split processor using Chainforge's splitText function.
  • Implemented the Python processor
    • Implemented the Python processor by reusing execute_python.
  • Wired the new processor implementations into worker.ts.
  • Testing the application on a cluster
    • Cluster tests (madagh server of DIRO): opened a tunnel (hole punch) to expose MySQL from my PC to madagh, then validated functionality. The application behaved as expected on the cluster.

Fourth Week

This week I fixed the bug of the input token calculation for an experiment input.

  • Input token calculation logic Fix
    • Context of the bug: Input token calculation always return 0
    • Causes
      • Our token calculator function have an if statement to check for final_dataset_id, if it's null we skip the calculation
      • But new experiments always have their final_dataset_id = null by default at the handle_save_template() function
      • Existing experiments have their result already processed so the calculator function just skip the calculation and returns 0
    • Solution
      • At the final_dataset_id check, we just go ahead and calculate the total input token
